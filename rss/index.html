2023-09-20T00:00:00Z 为什么要学习盲打和双拼 2023-09-20T00:00:00Z /blog/为什么要学习盲打和双拼/ <p>不知道你是否有这种情况：在打字时，不停低头看键盘，而且就用二个手指头打字（俗称“二指禅”），偶尔还用一下无名指，打字没啥规律；而且，打字好累阿，随便打个字，张（zhang），居然要按 5 下按键，按键盘都按半天；还要选字，一般给 7 个选项，看得头晕，还有下一页，一选字就要找半天，让人烦躁，以至于打字的速度和正确度都很低。</p> <p>简单总结一下，打字慢的原因：指法不标准（二指禅） + 有些字要按很多下按键（张，zhang） + 选字（给 7 个选项）</p> <p>这样的情况糟透了，很影响正常的聊天和学习时做笔记，甚至影响自己的思考，因为可能有时候你有一个灵感，但是打字速度太慢了，打字打到一半，灵感消失了......</p> <p>前阵子，我发现自己也有这种情况，所以，决定改变它。</p> <h2 id="%E8%A7%A3%E5%86%B3%E6%96%B9%E6%B3%95" tabindex="-1">解决方法</h2> <p>根据以上打字慢的原因，可以找到一些解决方法。</p> <p>指法不标准，没事，那就练标准阿，换句话说，学学盲打。</p> <p>有些字要按很多下按键，没事，那就用打任何字都只用按两下按键的输入法方案，换句话说，学学双拼。</p> <p>选字太累，要找好久，没事，那就用辅助码或形码，它们可以排除大部分候选字，换句话说，加辅助码或加形。</p> <h2 id="%E7%9B%B2%E6%89%93" tabindex="-1">盲打</h2> <p>盲打是一种不看键盘、用十根手指打字的标准打字方法，由于不用思考用哪个手指敲某个字母，可以显著提升打字速度，同时让打字体验更好。</p> <p>一图胜千言：</p> <p><img src="https://raw.githubusercontent.com/Zest-Zhang/blog-img/master/uTools_1673883517763.png" alt="1" /></p> <p>如图，这里指明了十个手指对应的键位，有一些细节要注意一下：</p> <ol> <li>打字前，左右手食指分别放在 F 键和 J 键上。</li> <li>打字后要复位：敲击键位后做好复位（手指回到 F 和 J）。复位的目的是根据 F、J 键快速找到其它键位。</li> <li>打字时减小力度、降低幅度后能打得更快。</li> <li>用左手大拇指敲击空格，当然，用右手大拇指敲击空格也行，两个拇指轮流用也行，看个人习惯吧。不过，个人觉得右手的小指负责的键位太多了，右手相对会累一点，所以用左手大拇指敲击空格，右手大拇指没怎么用。</li> </ol> <p>如果仔细看图片，可以发现双手的小指要负责的键位非常多，为什么小指要负责那么多键位？因为小指离那些键位最近......所以，练习的时候小指酸痛很正常，过几天就适应了。</p> <p>了解了手指对应的键位后，就要练习了，这个网站比较出名：<a href="https://www.keybr.com/%E3%80%82">https://www.keybr.com/。</a></p> <p>练习时是全英文的：</p> <p><img src="https://raw.githubusercontent.com/Zest-Zhang/blog-img/master/1695207187073.png" alt="1" /></p> <p>可以看打字时的平均速度：</p> <p><img src="https://img-blog.csdnimg.cn/58495c966af44ccea8d0c19ca9a99d8d.png?x-oss-process=image/watermark,type_d3F5LXplbmhlaQ,shadow_50,text_Q1NETiBA5oKf5Yid5aKD,size_20,color_FFFFFF,t_70,g_se,x_16" alt="speed" /></p> <p>不过，由于日常打字更多用中文，所以这个网站并不符合我的需求。经过搜索，我找到了一个适合练习中文和打字的软件：金山打字通。</p> <p><img src="https://dynamic-image.yesky.com/1200x-/uploadImages/2021/282/02/V7341K33YX3T.png" alt="金山打字通" /></p> <p>搜索金山打字通，在官网上下载一个，然后练习一个星期左右十个手指头应该就习惯了。</p> <h2 id="%E5%8F%8C%E6%8B%BC" tabindex="-1">双拼</h2> <p>双拼教程看这里：<a href="https://www.zhihu.com/question/383416202">教程</a>。</p> <p>里面有一句话讲的很好，讲明了使用双拼的好处：</p> <p>双拼能节省出大量时间去思考。一个动作重复 300 次，肌肉就会有记忆，重复 600 次 ，脊柱就会有记忆，大部分输入根本不用过大脑，直接脊柱反射就完成的时候，你的大脑就能完全无负担的思考，否则打个字都要停顿下的话，你会占用更多大脑 CPU 时间。</p> <p>教程讲得很清楚了，所以这篇文章没有必要写个双拼教程（不重复造轮子嘛）。</p> <p>看了教程后按这张图片练一天一般都会了：</p> <p><img src="https://picx.zhimg.com/80/v2-92168c5c3c42a0abef0fcbde2b06cf6d_1440w.webp?source=1940ef5c" alt="img" /></p> <p>双拼其实不难，作者就是修改了下键位的映射，把韵母的键位改了一下，声母不变。比如输入 zz，得到 走 字，这里第一个 z（声母） 不变，第二个 z（韵母） 不是 z ，而是被映射到 ou。</p> <p>另外，根据上面的教程，可以总结、补充下 iOS 的两种安装方法：</p> <ul> <li>在设置里搜索 双拼，然后选择 小鹤双拼 即可。</li> <li>app Store 搜索 iRime 输入法，然后选择 小鹤双拼，部署一下即可。</li> </ul> <p>不过，里面并没有说到辅助码，个人觉得如果不想因找字找得抓耳挠腮，可以学一下辅助码。</p> <h2 id="%E8%BE%85%E5%8A%A9%E7%A0%81" tabindex="-1">辅助码</h2> <p>什么是辅助码呢？以自然码为例（小鹤同理）做一演示：</p> <p>假设我们想打 音形，直接打 ynxy 是打不到的：</p> <p><img src="https://pica.zhimg.com/80/v2-beec6a19fa5e0411bb249244e496e22e_1440w.webp?source=1940ef5c" alt="img" /></p> <p>加个辅助码 l（立）、r（日） 就有了：</p> <p><img src="https://pic1.zhimg.com/80/v2-4b7537c30fbcafd4fd34064507c8890e_1440w.webp?source=1940ef5c" alt="img" /></p> <p>这里的 l 是 立 字的声母，r 是 日 的声母。</p> <p>单字也没有问题。举 字上面是 兴 字，直接加一个 x 就能找到字了：</p> <p><img src="https://pic1.zhimg.com/80/v2-9608d3595073b72ab33c4733a5a6aae0_1440w.webp?source=1940ef5c" alt="img" /></p> <h2 id="%E5%BD%A2%E7%A0%81" tabindex="-1">形码</h2> <p>什么是形码呢？根据笔画和部件（偏旁部首和其他）来编码的码。</p> <p>加形，其实就是用形码，五笔就是纯形码，小鹤音形是 音码 + 形码。</p> <p>前者学习成本太高，而且用了五笔就相当于双拼白学了，五笔是纯形码；后者看似还不错，实际有两个很大的问题：</p> <ol> <li>小鹤音形的词库是作者私人的，里面有一堆股票相关的词语，而且很多常用词都没有。</li> <li>紧接上个问题，你发现小鹤词库不好，就想自己改词库，但是，小鹤的词库是闭源的，闭源了就没法改词库。</li> </ol> <p>所以，二者都不建议，本人现在用的也不是二者，而是一个听起来就是键盘侠专用的 音码 + 形码 输入法方案，反正听起来不好听，所以对外称五笔终极版......</p> <p>至于是什么、怎么学，有空再说吧（或者 email me），文章内容挺多了已经......</p> <h2 id="%E6%80%BB%E7%BB%93" tabindex="-1">总结</h2> <p>如果不想折腾输入法方案、又想提高打字速度和改善打字体验，那么推荐：</p> <ul> <li>用金山打字通练习标准指法</li> <li>学习小鹤双拼（一天内可以学会）</li> <li>学习小鹤双拼里的辅助码</li> </ul> <p>以上就是盲打和双拼的介绍啦，感谢阅读~</p> git的常用代码和使用场景 2023-09-04T00:00:00Z /blog/git常用代码及使用场景/ <p>本篇文章并不是 git 操作的全面科普，而是对 git 常用操作的代码和使用场景进行总结。</p> <p>以下是 git 的常用代码和使用场景：</p> <h2 id="%E4%B8%8A%E4%BC%A0%E6%96%B0%E4%BB%93%E5%BA%93" tabindex="-1">上传新仓库</h2> <p>使用场景：在桌面新建项目，想把这个项目上传到 github 上的新仓库时。</p> <p>代码：</p> <pre class="language-sh"><code class="language-sh"><span class="token comment"># 右键 git bash 需要上传的目录，依次输入以下代码：</span><br /><span class="token function">git</span> init<br /><span class="token function">git</span> <span class="token function">add</span> <span class="token builtin class-name">.</span><br /><span class="token function">git</span> status<br /><span class="token function">git</span> commit <span class="token parameter variable">-m</span> <span class="token string">"注释"</span><br /><span class="token function">git</span> remote <span class="token function">add</span> origin url<br /><span class="token function">git</span> push origin master</code></pre> <p>根据需要修改 commit message（就是 <code>注释</code> 二字），url 则是在 github 新建仓库后复制一下即可。</p> <h2 id="%E4%BD%BF%E7%94%A8git-push%E4%B8%8A%E4%BC%A0" tabindex="-1">使用git push上传</h2> <p>使用场景：master 分支使用 git push 上传（而不是 git push origin master）</p> <p>代码：</p> <pre class="language-sh"><code class="language-sh"><span class="token function">git</span> push --set-upstream origin master</code></pre> <p>它的作用很明显，可以少打两个单词。</p> <p>如果有其他分支也想这样做，比如 gh-pages 分支，假设已经新建了 gh-pages 分支，使用以下代码：</p> <pre class="language-sh"><code class="language-sh"><span class="token function">git</span> checkout gh-pages<br /><span class="token function">git</span> push --set-upstream origin gh-pages</code></pre> <p>以后先用 <code>git checkout gh-pages</code> 切换到 gh-pages 分支，再使用 git push 即可。</p> <h2 id="git-%E6%8F%90%E4%BA%A4%E8%B6%85%E6%97%B6" tabindex="-1">git 提交超时</h2> <p>使用场景：出现 <code>Failed to connect to github.com port 443 after 21074 ms: Timed out</code> 的超时错误</p> <p>代码：</p> <pre class="language-sh"><span class="token comment"># 获取历史 commit 记录</span><br /><span class="token function">git</span> reflog<br /><span class="token comment"># HEAD 改成想要回溯到的 commit</span><br /><span class="token function">git</span> reset HEAD<br /><span class="token function">git</span> push origin master <span class="token parameter variable">-f</span></code></pre> <p>需要注意，<code>git reset</code> 个人用可以，团队配合用 <code>git revert</code> 更好，原因看演示：</p> <p>当前处于 C2 这个 commit 状态上，使用 <code>git revert C2</code>：</p> <p><img src="https://raw.githubusercontent.com/Zest-Zhang/blog-img/master/uTools_1679408783870.png" alt="uTools_1679408783870" /></p> <p>奇怪，撤销的 commit 记录后面居然多了一个新 commit ，原因：新 commit 记录 <code>C2'</code> 引入了更改，这些更改刚好是用来撤销 <code>C2</code> 这个 commit 的，也就是说，<code>C2'</code> 的状态与 <code>C1</code> 是相同的，revert 之后就可以把更改推送到远程 repo 与别人分享了。</p> <p>不过，revert 的 commit 参数和 reset 不太一样，reset 的参数使用后就回到了参数对应的那个 commit 记录，revert 参数使用后回到了该参数的上一 个 commit 记录，而且不会影响前面的 commit 记录，reset 使用后前面的 commit 记录会消失，用 <code>git log</code> 查询不到，可以用 <code>git reflog</code> 查询。</p> <h2 id="feature%E5%88%86%E6%94%AF" tabindex="-1">feature分支</h2> <p>使用场景：开发一个新功能就新建 feature 分支，避免把 dev 分支搞乱，在 feature 上开发，完成后，新功能开发好就把 feature 分支合并到开发分支 dev，最后，删除 feature 分支。</p> <p>代码：</p> <pre class="language-sh"><code />checkouted to a new branch <span class="token string">'feature-vulcan'</span></code></pre> <pre class="language-sh"><code class="language-sh"><span class="token comment"># vulcan 功能开发好了</span><br /><span class="token function">git</span> <span class="token function">add</span> vulcan.js<br /><span class="token function">git</span> status<br />On branch feature-vulcan<br /> new file: vulcan.js<br /><br /><span class="token function">git</span> commit <span class="token parameter variable">-m</span> <span class="token string">"add feature vulcan"</span></code></pre> <pre class="language-sh"><code class="language-sh"><span class="token comment"># 合并到开发分支 dev，然后删除 feature、切换到 `dev`：</span><br /><span class="token function">git</span> rebase dev<br /><span class="token function">git</span> branch <span class="token parameter variable">-d</span> feature-vulcan<br /><span class="token function">git</span> checkout dev</code></pre> <p>如果突然接到上级命令，因为 xx 原因，新功能取消，所以在 feature 没合并到 dev 时强行删除 feature：</p> <pre class="language-sh"><code class="language-sh"><span class="token function">git</span> branch <span class="token parameter variable">-D</span> feature-vulcan</code></pre> <p>（使用小写的参数 <code>-d</code> 会删除失败，因为 feature-vulcan 分支还没被合并，如果删除，将丢失掉修改）</p> <h2 id="bug%E5%88%86%E6%94%AF" tabindex="-1">bug分支</h2> <p>使用场景：每个 bug 都可以通过新建一个 bug 分支修复，修复后，合并分支，然后将临时分支删除。所以，当你接到任务（修复一个代号 101 的 bug）时，很自然地，你想创建一个分支 <code>issue-101</code> 来修复它。但是，当前正在<code>dev</code>上进行的工作还没有提交，不是你不想提交，而是工作只进行到一半，还没法提交，预计完成还需 1 天时间。此时，要在两个小时内修复这个 bug，如何处理？</p> <p>代码：</p> <pre class="language-sh"><code class="language-sh"><span class="token comment"># 前提：在 dev 上工作且生产分支是 master，bug 来自生产分支 master</span><br /><span class="token comment"># 使用 git stash 保存 dev 上进行到一半的工作</span><br /><span class="token function">git</span> stash<br /><span class="token comment"># 先确定要在哪个分支上修复 bug，假定需要在 master 分支上修复，就在 master 分支上创建 bug 分支</span><br /><span class="token function">git</span> checkout master<br /><span class="token function">git</span> checkout <span class="token parameter variable">-b</span> issue-101</code></pre> <pre class="language-sh"><code class="language-sh"><span class="token comment"># 修复 bug：把“git is free software ...”改为“git is a free software ...”，然后提交</span><br /><span class="token function">git</span> <span class="token function">add</span> readme.txt <br /><span class="token function">git</span> commit <span class="token parameter variable">-m</span> <span class="token string">"fix bug 101"</span></code></pre> <pre class="language-sh"><code class="language-sh"><span class="token comment"># 修复完成后，切换到 master 分支，并完成合并，最后删除 issue-101 分支</span><br /><span class="token function">git</span> checkout master<br /><span class="token function">git</span> merge --no-ff <span class="token parameter variable">-m</span> <span class="token string">"merged bug fix 101"</span> issue-101<br /><span class="token comment"># bug 已经修复，继续回 dev 分支工作</span><br /><span class="token function">git</span> checkout dev<br /><span class="token comment"># 恢复 dev 上进行到一半的工作并删除 stash 记录</span><br /><span class="token function">git</span> stash pop</code></pre> <p>在 master 上修复 bug 后，由于 dev 是从 master 上分出来的，所以，这个 bug 其实在当前 dev 分支上也存在。</p> <p>如何在 dev 上修复同样的 bug？</p> <p>git 提供了 <code>cherry-pick</code> 命令，它能复制一个特定的提交到当前分支：</p> <pre class="language-sh"><code class="language-sh"><span class="token comment"># 把在 issue-101 分支上的提交记录 4c805e2 复制到 dev 分支</span><br /><span class="token function">git</span> checkout dev<br /><span class="token function">git</span> cherry-pick 4c805e2</code></pre> <p>使用 <code>git cherry-pick</code> 后，就不用在 dev 分支上把修 bug 的过程再重复一遍了。</p> <p>cherry-pick 实战：</p> <ul> <li>gh-pages 上有两个文件： <img src="https://raw.githubusercontent.com/Zest-Zhang/blog-img/master/uTools_1693813494285.png" alt="1" /></li> <li>切换到 master 分支（只有一个文件）： <img src="https://raw.githubusercontent.com/Zest-Zhang/blog-img/master/uTools_1693813521925.png" alt="2" /></li> <li>把 gh-pages 提交第二个文件时的 commit 号复制一下，然后使用 cherry-pick（发现 master 分支上也有第二个文件了）： <img src="https://raw.githubusercontent.com/Zest-Zhang/blog-img/master/uTools_1693813576650.png" alt="3" /></li> </ul> <h2 id="git-stash%E4%BF%9D%E5%AD%98%E6%9C%AAcommit%E7%9A%84%E4%BB%A3%E7%A0%81" tabindex="-1">git stash保存未commit的代码</h2> <p>使用场景：某一天，你正在 feature 分支开发新需求，突然产品经理跑过来说线上有 bug，必须马上修复。此时，你在 dev 分支上开发功能，开发到一半，于是你想切到 master 分支，就会看到以下报错：</p> <p><img src="https://raw.githubusercontent.com/Zest-Zhang/blog-img/master/68f81dbc24944538bf63d8530f62cc2a%7Etplv-k3u1fbpfcp-zoom-in-crop-mark_1304_0_0_0.webp" alt="img" /></p> <p>因为当前有文件更改了，需要提交 commit /blog/为什么做这个博客/ <p>尽管已经搭建过很多博客了（hexo、vuepress、11ty），但前面的博客搭建效果并不满意，因此迟迟没有开始写文章。</p> <p>现在的这个博客还算满意，极简的风格，符合自己的价值观，所以就开始想写文章了，这是本博客的第一篇文章，希望能写好~</p> <p>以下是我做这个博客的原因：</p> <h2 id="%E7%94%A8%E6%9D%A5%E4%BB%8B%E7%BB%8D%E8%87%AA%E5%B7%B1" tabindex="-1">用来介绍自己</h2> <p>你可以把 你是怎样的人、想和怎么样的人共事、未来想做什么 等内容都放里面，如果别人（比如 朋友、hr、面试官）想了解现在的你，把博客链接发给他们就可以了。</p> <p>博客更偏向于让朋友了解我是什么样的人和我想做什么（比如对交流的要求）。</p> <h2 id="%E7%BB%B4%E6%8A%A4%E4%B8%80%E4%B8%AA%E5%8D%9A%E5%AE%A2%E4%B8%8D%E6%98%AF%E4%B8%80%E4%BB%B6%E9%9A%BE%E4%BA%8B" tabindex="-1">维护一个博客不是一件难事</h2> <p>维护一个博客不是一件难事，可能刚开始建站时比较麻烦，但维护起来不需要付出太多（域名费用 + 写文章的时间精力成本），而且可以不限篇幅，不限频率。</p> <h2 id="%E6%83%B3%E6%8A%8A%E7%9F%A5%E8%AF%86%E8%A1%A8%E8%BE%BE%E5%BE%97%E6%9B%B4%E6%B8%85%E6%A5%9A" tabindex="-1">想把知识表达得更清楚</h2> <p>道可道，非常道。很多技术教程里的东西表述得很清楚，但在实战的时候会遇到很多问题，比如 忽略了某个细节、内容表述容易引起误解，因为某样知识或技能的细节也许教程的作者都清楚了，但做教程的时候因为各种原因没说，比如 忘记说了、认为这个细节太简单就不说了，这会给初学者造成很大的困扰，进而走了很多弯路，浪费了很多时间；如果有一个博客，自己可以把细节都记下来，把想表达的某样知识表达得更清楚，那学起来应该会更轻松吧；听起来似乎有点像读书笔记，不过实际上应该不只是读书笔记，看看下面内容就明白了。</p> <p>除此之外，好的文章也可以从侧面证明自己的沟通能力。</p> <h2 id="%E5%88%86%E4%BA%AB%E7%94%9F%E6%B4%BB%E7%BB%8F%E9%AA%8C%E5%92%8C%E4%B8%AA%E4%BA%BA%E6%83%B3%E6%B3%95" tabindex="-1">分享生活经验和个人想法</h2> <p>如果有一个博客，陌生人通过博客了解到你是怎么样的人、你有什么灵感、你做过什么，比如 正准备找工作，一位 HR（陌生人）要了解你，一份简历很显然不足以让他了解你的价值，如果你有一个精彩的博客，HR 也许能通过你的博客了解你对技术的属于自己的思考、你在业余时间在想什么，它能让陌生人了解你是什么样的人。</p> <p>这个博客除了技术内容，还有更多的生活经验和个人想法。人是懒惰的，而技术文章也没那么好写，所以，只写技术文章大概率不会坚持写博客，因为接触过的只写技术文章的博主的博客基本上都已经好几年不更新了；希望自己既有技术文章，也有生活经验文章，文章都以实用为主，没有碎碎念或乱七八糟的吐槽，因此更新不会很频繁，不过应该不会停止更新，毕竟，一个博客的死掉不是不再更新，而是无法访问。</p> <h2 id="%E8%83%BD%E5%A4%9F%E4%BA%A4%E5%88%B0%E5%BF%97%E5%90%8C%E9%81%93%E5%90%88%E7%9A%84%E6%9C%8B%E5%8F%8B" tabindex="-1">能够交到志同道合的朋友</h2> <p>为什么博客在结交志同道合的朋友方面的潜力要远胜于原始的交谈方式？</p> <ul> <li>博客无地域限制，整个互联网上从 A 到 B 只有一个点击的距离，而传统的建立朋友圈子的方法则受到地域限制</li> <li>如果按照以前结交朋友的方式，需要互相聊天，交流观点，然后才逐渐熟悉起来，但这需要一个较长的过程，而且更糟糕的是，当你遇到另一个陌生人，又要把整个过程重复一次，表达你已经对老友表达过的那番想法。可博客却做到了“一次表达，无数次阅读”，当我看到一个写了好几年的博客，看完了之后我仿佛和这个人交谈了很久</li> </ul> <p>以上就是写这个博客的原因啦，感谢阅读~</p>
